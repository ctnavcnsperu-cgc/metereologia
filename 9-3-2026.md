### 🟢 Consulta: Implementación del TAF (Pronóstico de Aeródromo)

A raíz de la necesidad de armar nuevas tablas con pronósticos, se ha realizado una investigación técnica sobre la obtención de datos **TAF**. Estos son los hallazgos:

#### 1. Disponibilidad del Dato
Actualmente el sistema solo extrae **METAR** (tiempo actual). El **TAF** no está siendo capturado pero es totalmente factible obtenerlo sin afectar la producción actual.
*   **Endpoint Identificado**: `https://meteorologia.corpac.gob.pe/app/Meteorologia/pronosticos/manualTaf.php`
*   **Parámetro de Consulta**: Se usa `aeropT` con el código OACI (ej: `SPJC`) mediante una petición POST.

#### 2. Mecanismo de Obtención
El proceso es idéntico al del METAR, lo que facilita una integración limpia:
*   El servidor devuelve un bloque HTML que contiene el texto del TAF ("TAF SPJC 091100Z...").
*   Se requiere una función de procesamiento para limpiar el texto y extraer periodos de validez.

#### 3. Impacto en el Rendimiento (Scraping)
Se evaluó si esto ralentizaría el proceso actual (31 aeropuertos con pausas de 1.2s):
*   **Sin Optimizar**: Si se hace una consulta extra por cada aeropuerto, el tiempo total pasaría de **~53 segundos a ~106 segundos**.
*   **Optimización Posible**: El servidor permite consultas por lotes de hasta **4 aeropuertos** y una opción (`incMetar=on`) para traer ambos datos (METAR+TAF) en una sola respuesta. Esto podría reducir drásticamente las peticiones al servidor.

#### 4. Recomendación de Seguridad
Para no "romper" la aplicación en producción, se recomienda:
1.  **Escritura de un Script de Prueba**: Crear `test_taf.py` para validar la captura del texto sin tocar el `scraper.py` principal.
2.  **Validación de Formato**: Asegurar que el bloque de texto del TAF sea legible en las nuevas tablas propuestas.

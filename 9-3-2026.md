### 🟢 Consulta: Implementación del TAF (Pronóstico de Aeródromo)

A raíz de la necesidad de armar nuevas tablas con pronósticos, se ha realizado una investigación técnica sobre la obtención de datos **TAF**. Estos son los hallazgos:

#### 1. Disponibilidad del Dato
Actualmente el sistema solo extrae **METAR** (tiempo actual). El **TAF** no está siendo capturado pero es totalmente factible obtenerlo sin afectar la producción actual.
*   **Endpoint Identificado**: `https://meteorologia.corpac.gob.pe/app/Meteorologia/pronosticos/manualTaf.php`
*   **Parámetro de Consulta**: Se usa `aeropT` con el código OACI (ej: `SPJC`) mediante una petición POST.

#### 2. Mecanismo de Obtención
El proceso es idéntico al del METAR, lo que facilita una integración limpia:
*   El servidor devuelve un bloque HTML que contiene el texto del TAF ("TAF SPJC 091100Z...").
*   Se requiere una función de procesamiento para limpiar el texto y extraer periodos de validez.

#### 3. Impacto en el Rendimiento (Scraping)
Se evaluó si esto ralentizaría el proceso actual (31 aeropuertos con pausas de 1.2s):
*   **Sin Optimizar**: Si se hace una consulta extra por cada aeropuerto, el tiempo total pasaría de **~53 segundos a ~106 segundos**.
*   **Optimización Posible**: El servidor permite consultas por lotes de hasta **4 aeropuertos** y una opción (`incMetar=on`) para traer ambos datos (METAR+TAF) en una sola respuesta. Esto podría reducir drásticamente las peticiones al servidor.

#### 4. Recomendación de Seguridad
Para no "romper" la aplicación en producción, se recomienda:
1.  **Escritura de un Script de Prueba**: Crear `test_taf.py` para validar la captura del texto sin tocar el `scraper.py` principal.
2.  **Validación de Formato**: Asegurar que el bloque de texto del TAF sea legible en las nuevas tablas propuestas.

### 🔵 Consulta: Frecuencia de Actualización (Cron de 3 vs 5 minutos)

**Usuario:** ¿crees con con el tiempo que tarda en hacer scraping se pueda usat un cron de 3 minutos en lugar de 5? comentame

**Respuesta:** La respuesta corta es: **Sí, es totalmente posible**, pero con la optimización de lotes (batching) se vuelve mucho más seguro y eficiente.

Aquí te detallo el análisis técnico para bajar de 5 a 3 minutos:

#### 1. El factor del "Tiempo de Descanso" del Servidor
Lo que más molesta a un servidor (y lo que provoca bloqueos de IP) no es solo cuántas veces entras al día, sino el **ritmo** de las peticiones.

*   **Sin Optimizar (Individual):** El script tarda casi 2 minutos de los 3 disponibles. Estarías ocupando el servidor el 66% del tiempo. Esto "se siente" como un ataque constante.
*   **Con Optimización (Lotes de 4):** El script terminaría en unos 15 a 20 segundos. Esto significa que el servidor de CORPAC solo recibe tráfico tuyo durante el 10% del ciclo. El resto del tiempo (2 min 40 seg) no hay actividad, lo cual es muy "educado" desde el punto de vista de scraping.

#### 2. ¿Realmente se actualiza la información cada 3 minutos?
Este es el punto clave. Los datos meteorológicos aeronáuticos tienen ciclos:
*   **METAR:** Se suele emitir cada 30 minutos o cada hora (ej: a las :00 y a las :30). Solo se emite un "SPECI" (extraordinario) si el clima cambia bruscamente.
*   **TAF:** Se actualiza cada 6 horas por lo general.

**¿Qué ganamos con 3 minutos?** Ganamos que, si CORPAC publica el reporte de las 14:00Z a las 14:02Z, tú lo tendrás en tu pantalla a las 14:03Z en lugar de esperar hasta las 14:05Z o 14:07Z. Es una ganancia de **rapidez de captura**, no de nuevos datos.

#### 3. Riesgos y cómo evitarlos
Si decides bajar a 3 minutos, te sugiero considerar lo siguiente:

1.  **Evitar Solapamiento:** Si por alguna razón el servidor de CORPAC está lento y el script de las 12:00 tarda 4 minutos, el cron lanzará otro script a las 12:03 mientras el primero sigue corriendo. Esto podría causar un efecto de "bola de nieve". 
    *   *Solución:* Usar un archivo de bloqueo (`.lock`) que impida que el script se ejecute si ya hay uno corriendo.
2.  **User-Agent Rotativo:** Para ser más indetectables, podríamos hacer que el script "se identifique" como diferentes navegadores en cada ejecución.
3.  **Pausas Aleatorias:** En lugar de una pausa fija de 1.2s, podemos usar `time.sleep(random.uniform(1.0, 2.0))` para que el tráfico parezca humano y no un bot matemático.

**Opinión profesional:** Si implementamos la **Consulta por Lotes (Batching)**, bajar a **3 minutos es totalmente seguro**. Sin batching, me quedaría en 5 minutos para evitar que el servidor de CORPAC te marque como una "carga pesada" persistente.
